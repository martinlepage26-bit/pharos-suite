# Method Rules

## Core claim discipline

Every major paragraph should answer three questions:
1. What is the claim?
2. What mechanism supports it?
3. What consequence domain does it affect?

Do not skip the mechanism layer.

## Artifact classes

Use these classes unless the packet requires refinement:

### 1. Source-bearing artifact
Material that can directly support factual, procedural, or documentary claims.
Examples: notes, logs, primary manuscripts, direct records, standards, statutes, archived correspondence.

### 2. Generated synthesis artifact
Material that compresses, reframes, compares, rewrites, or editorializes other materials.
Examples: revised drafts, summary packets, comparison documents, AI rewrites, generated clean copies.

### 3. Control artifact
Material that specifies, audits, or encodes workflow constraints.
Examples: method statements, governance worksheets, decision rules, review gates, control tables.

### 4. Visualization artifact
Material that renders structure but does not, by itself, settle factual disputes.
Examples: storyboards, diagrams, dashboards, slide decks, HTML visual summaries.

## Recursive tests

### Source-tree test
What in this packet is closest to first-order evidence?
What materials are downstream of it?
Do not let downstream artifacts silently replace the source tree.

### Re-entry test
Which derived artifact has re-entered the workflow as if it were source?
If this happens, mark it as governance-on-governance risk.

### Admissibility test
What was allowed into the loop at each stage?
What was blocked, excluded, or newly admitted?
If these rules changed, state that the governing object changed too.

### Method-lock test
Has one editorial or procedural path hardened into the only visible account of the archive?
Signs include repeated plateau counts, repeated summary shapes, disappearance of competing branches, or clean language masking unstable inputs.

### Interruption test
Where could a reviewer, editor, operator, or affected party reopen the claim?
If there is no point of practical interruption, the workflow has an accountability problem.

### Escalation test
Did the next recursive pass increase explanatory power, or did it only add polish?
If it only adds polish, stop the recursion and analyze the current layer.

## Control extraction rules

A valid control must name:
- the failure or risk pattern
- the workflow interface where it attaches
- the accountable actor or role
- the evidence artifact required to satisfy it
- the review interval or trigger

Weak controls to avoid:
- “be transparent”
- “add oversight”
- “document better”

Strong controls are interface-bound and testable.

## Authorship and disclosure rule

Use three functional layers:
- **clerical mediation**: spelling, formatting, citation cleanup, grammar
- **structural mediation**: reorganization, flow, comparison, paragraph sequencing, coherence pressure
- **epistemic mediation**: proposing distinctions, identifying overreach, surfacing contradictions, suggesting mechanisms, testing analytic scope

When the packet shows epistemic mediation, do not describe the AI role as “proofreading only.”

## Stop rules

Stop and write the analysis when any of these are true:
- the next pass repeats the same architecture
- findings are stable across artifact classes
- the remaining uncertainty is evidentiary rather than interpretive
- additional synthesis would further flatten the source hierarchy
