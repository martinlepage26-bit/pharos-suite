prompt engineering vs context engineering
While both aim to improve AI outputs, prompt engineering focuses on the specific instructions you give a model, whereas context engineering is about designing the entire information environment the model sees.

The industry has largely shifted toward context engineering because modern AI systems (especially autonomous agents) rely less on "clever wording" and more on having the right data at the right time.

Core Comparison
Feature 	Prompt Engineering	Context Engineering
Primary Goal	Writing better instructions	Building better information systems
Focus	How you phrase the request	What the model "knows" when it responds
Method	Manual trial-and-error (e.g., "Think step by step")	Programmatic curation of data, memory, and tools
Scope	One-off, single-turn interactions	Multi-turn, state-aware systems and agents
Mindset	Copywriter or instructional designer	Systems architect or software engineer
Key Components of Context Engineering
Context engineering is often described as the "macro-architecture" that surrounds a prompt. It involves several layers:

 +1
Knowledge Retrieval (RAG): Dynamically pulling relevant facts from databases or the web so the model doesn't have to rely on its training data alone.
Memory Management: Maintaining a "short-term" conversation history and "long-term" user preferences to provide continuity across sessions.
Tool Orchestration: Defining which APIs or functions (like a calculator or calendar) are available to the AI and how their results flow back into the conversation.
Context Pruning: Actively removing irrelevant tokens to prevent "context rot," where a model loses focus because its input window is too cluttered.

Which one should you use?
Use Prompt Engineering for simple tasks where the model already has all necessary information (e.g., summarizing a short paragraph or changing the tone of an email).
Use Context Engineering for production-grade applications that require reliability, accuracy, and the ability to handle complex, multi-step workflows.

Implementation Strategies
1. Optimization for Prompt Engineering
Few-Shot Examples: Include 1–2 "Input/Output" pairs within the instructions to show the model exactly what a perfect response looks like.
Negative Constraints: Explicitly list what the AI should not do (e.g., "Do not use deprecated libraries" or "Never hard-code API keys").
Action Verbs: Use precise verbs like "Extract," "Analyze," or "Synthesize" instead of vague terms like "Help with".

Implementation Tips
Trigger Keywords: Ensure the description field includes the exact terms that should activate the skill automatically.
Token Efficiency: Keep the body under 500 lines. If you have massive reference lists, move them to a separate file and tell the AI to "Read [file] only if necessary".
Tool-First Logic: Instead of describing how to do a calculation, provide a tool or script the AI can call. This ensures 100% accuracy and saves context space.

How to distinguish the two in your file:
Engineering Type	Implementation in SKILL.md
Prompt (The "How")	Using XML tags (<instructions>) and Persona settings to control the model's behavior and linguistic style.
Context (The "Where")	Creating a Map of your file system and telling the AI exactly which files to read for specific facts (minimizing "hallucinations").
Pro-Tip for Context Engineering
Don't paste your entire codebase into the skill. Instead, provide a Table of Contents of your project. This allows the AI to "fetch" the right context only when it needs it, keeping the window clean and the responses fast.

<logic_flow>

3. **Format:** Always return results in [Markdown Table/JSON/Code].
</logic_flow>

<negative_constraints>
- NEVER use conversational fluff ("As an AI...", "I hope this helps").
- DO NOT provide solutions for [Out-of-Scope Topic].
</negative_constraints>

# 2. THE MAP (Context Engineering)
<!-- Focus: Where to find data and how to use tools -->

<data_environment>
- **Local Truth:** Always prioritize facts in `/docs/specs.md` over internal training.
- **Pattern Matching:** Reference `/templates/boilerplate.ts` for all new files.
- **Token Pruning:** If the task is [X], ignore all files in `/archive/`.
</data_environment>

<tool_orchestration>
- **Verification:** Run `scripts/verify.py` before finalizing any logic.
- **External Bridge:** Use the `fetch_api` tool if `data/local_cache.json` is > 6 hours old.
</tool_orchestration>

<memory_strategy>
- **User Context:** Read `.ai/user_preferences.yaml` for custom naming conventions.
- **Session State:** Summarize current progress every 5 turns to prevent "context drift."
</memory_strategy>

Why this works:
Prompt Engineering (The "Software"): The XML tags and persona logic ensure the model's reasoning is sharp. It prevents the AI from being "lazy" or "chatty."
Context Engineering (The "Hardware"): The directory mapping and tool rules ensure the model has the right tools and data. It solves the "hallucination" problem by giving it a physical file path to follow.
The "Pro" Secret
If your project is huge, do not put data inside the SKILL.md. Use the Context Engineering section to give the AI a "Table of Contents" of your file system. This forces it to search and fetch only what it needs, keeping your context window clean.
